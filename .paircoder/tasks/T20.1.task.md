# T20.1 — Parts API

## Mantra: Each function IS a car part

Create `engine/parts_api.py` — defines the 10 car parts that players code. Each part is a function with a specific signature, valid output ranges, and a working default. Hardware specs set the physical limits.

## The 10 Parts (CAR_PARTS)

| # | Part | What It IS | Player Codes | Output |
|---|------|-----------|-------------|--------|
| 1 | engine_map | The engine's brain | How it responds to throttle at each RPM | (torque_pct, fuel_flow_pct) |
| 2 | gearbox | The transmission | When to shift up/down | target_gear (1-8) |
| 3 | ers_deploy | The battery deploy system | When to dump stored energy | deploy_kw (0-120) |
| 4 | ers_harvest | The regeneration system | How hard to recover under braking | harvest_kw (0-120) |
| 5 | brake_bias | The brake balance bar | Front vs rear braking split | front_pct (50-65) |
| 6 | suspension | The ride height controller | How low to run for ground effect | ride_height (-1 to 1) |
| 7 | cooling | The cooling duct controller | How much airflow to sacrifice for cooling | cooling_effort (0-1) |
| 8 | fuel_mix | The fuel mixture valve | Rich (power) vs lean (saving) | lambda (0.85-1.15) |
| 9 | differential | The diff lock controller | Traction vs rotation in corners | lock_pct (0-100) |
| 10 | strategy | The pit wall radio | When to pit, what compound | dict |

## Hardware Specs (the physical limits your code operates within)

ENGINE_SPEC, AERO_SPEC, CHASSIS_SPEC — player picks hardware, then codes how it behaves.
